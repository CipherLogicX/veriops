# TrackQA — Security (v1)

## Authentication

- Email + password login via OAuth2 password flow (`POST /api/v1/auth/login`).
- Passwords hashed with bcrypt (`passlib`); plaintext is never stored or logged.
- JWT access tokens (default 60 min) and refresh tokens (default 7 days), signed with
  `JWT_SECRET_KEY` (HS256). Generate a strong secret: `openssl rand -hex 32`.
- No users are hardcoded. The first admin is created on first boot from
  `FIRST_ADMIN_EMAIL` / `FIRST_ADMIN_PASSWORD`. The bootstrap is idempotent.
- Token storage: the SPA keeps tokens in `localStorage` for v1. This is an accepted
  trade-off for an internal tool; moving to httpOnly cookies is a known later step.

## Authorization (RBAC)

System roles, seeded on boot: `SUPER_ADMIN`, `ORG_ADMIN`, `INTEGRATION_ADMIN`
(admin), and `PROJECT_MANAGER`, `QA_LEAD`, `TESTER`, `DEVELOPER`, `VIEWER`,
`AUDITOR` (non-admin).

Enforcement points (backend, never trust the client):
- `get_current_user` — every protected route resolves the user and role set from the token.
- `require_admin` — admin-only routes (all of `/api/v1/admin/*`) reject non-admins with 403.
- `require_project_access` — project-scoped routes require membership; admins bypass.

Frontend mirrors this for UX only: `RoleGuard` hides the Admin Console and redirects
non-admins. The backend remains the source of truth — hiding a link is not a control.

## Row/Tenant scoping (RLS approach)

v1 enforces scoping in the service/query layer: every list/read is filtered by
`organization_id` and, for project data, by project membership. This is application-level
row security. The schema (organization_id on every important table) is laid out so that
PostgreSQL native RLS policies can be added later without data migration.

## Secrets & configuration

- All secrets come from environment variables (`.env`), never from code.
- `.env` is gitignored; `.env.example` documents required keys with safe placeholders.
- Integration credentials are intentionally NOT stored in v1. The `integrations` table
  holds non-secret config only; a future `integration_credentials` table will store
  encrypted values, managed by admins exclusively.

## Transport security

- Edge nginx terminates TLS; all HTTP is redirected to HTTPS (except `/health`).
- v1 ships a self-signed certificate generated by `setup.sh`. Replace
  `deployment/nginx/ssl/trackqa.{crt,key}` with CA-issued certs for real deployments.
- Basic security headers set at the edge: `X-Frame-Options`, `X-Content-Type-Options`,
  `Referrer-Policy`.

## Error handling & logging

- Clients never receive stack traces. Unhandled errors return a generic message plus a
  `request_id`.
- The server logs technical detail (route, method, status, duration, request_id) and full
  stack traces server-side only.
- Security-relevant actions (user create, project create, executions, defect changes) are
  written to `audit_logs` and visible to admins under Admin Console → Audit Logs.

## Known v1 limitations

- Rate limiting is not yet enforced (Redis is wired for it).
- No SSO/LDAP yet (roles enumerate them as planned connectors only).
- Token revocation is time-based (expiry) rather than a server-side denylist.
